# <a name="microservices-architecture"></a><span data-ttu-id="b2036-103">Architettura di microservizi</span><span class="sxs-lookup"><span data-stu-id="b2036-103">Microservices architecture</span></span>

<span data-ttu-id="b2036-104">Come suggerisce il nome, un'architettura di microservizi è un approccio alla creazione di un'applicazione server come un set di piccoli servizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-104">As the name implies, a microservices architecture is an approach to building a server application as a set of small services.</span></span> <span data-ttu-id="b2036-105">Ciò significa che un'architettura di microservizi è principalmente orientata al back-end, anche se l'approccio viene usato anche per il front-end.</span><span class="sxs-lookup"><span data-stu-id="b2036-105">That means a microservices architecture is mainly oriented to the back-end, although the approach is also being used for the front end.</span></span> <span data-ttu-id="b2036-106">Ogni servizio viene eseguito nel rispettivo processo e comunica con gli altri processi tramite protocolli quali HTTP/HTTPS, WebSocket o [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span><span class="sxs-lookup"><span data-stu-id="b2036-106">Each service runs in its own process and communicates with other processes using protocols such as HTTP/HTTPS, WebSockets, or [AMQP](https://en.wikipedia.org/wiki/Advanced_Message_Queuing_Protocol).</span></span> <span data-ttu-id="b2036-107">Ogni microservizio implementa una funzionalità aziendale o un dominio end-to-end specifico all'interno di un determinato limite di contesto. Deve essere inoltre sviluppato in modo autonomo ed essere distribuito in modo indipendente.</span><span class="sxs-lookup"><span data-stu-id="b2036-107">Each microservice implements a specific end-to-end domain or business capability within a certain context boundary, and each must be developed autonomously and be deployable independently.</span></span> <span data-ttu-id="b2036-108">Ogni microservizio, infine, deve essere proprietario del modello di dati di dominio e della logica di dominio corrispondenti (sovranità e gestione decentralizzata dei dati), che possono essere basati su varie tecnologie di archiviazione dati (SQL, NoSQL) e linguaggi di programmazione diversi.</span><span class="sxs-lookup"><span data-stu-id="b2036-108">Finally, each microservice should own its related domain data model and domain logic (sovereignty and decentralized data management) and could be based on different data storage technologies (SQL, NoSQL) and different programming languages.</span></span>

<span data-ttu-id="b2036-109">Quali dimensioni deve avere un microservizio?</span><span class="sxs-lookup"><span data-stu-id="b2036-109">What size should a microservice be?</span></span> <span data-ttu-id="b2036-110">Quando si sviluppa un microservizio, le dimensioni non hanno molta importanza.</span><span class="sxs-lookup"><span data-stu-id="b2036-110">When developing a microservice, size shouldn't be the important point.</span></span> <span data-ttu-id="b2036-111">È importante invece creare servizi a regime di controllo libero, in modo da avere, per ogni servizio, piena autonomia di sviluppo, distribuzione e scalabilità.</span><span class="sxs-lookup"><span data-stu-id="b2036-111">Instead, the important point should be to create loosely coupled services so you have autonomy of development, deployment, and scale, for each service.</span></span> <span data-ttu-id="b2036-112">Quando si identificano e si progettano microservizi, naturalmente, è consigliabile crearli più piccoli possibile, senza tuttavia avere un numero eccessivo di dipendenze dirette da altri microservizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-112">Of course, when identifying and designing microservices, you should try to make them as small as possible as long as you don't have too many direct dependencies with other microservices.</span></span> <span data-ttu-id="b2036-113">Più delle dimensioni, infatti, sono importanti la coesione interna del microservizio e l'indipendenza dagli altri servizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-113">More important than the size of the microservice is the internal cohesion it must have and its independence from other services.</span></span>

<span data-ttu-id="b2036-114">Perché un'architettura di microservizi?</span><span class="sxs-lookup"><span data-stu-id="b2036-114">Why a microservices architecture?</span></span> <span data-ttu-id="b2036-115">In breve, offre flessibilità a lungo termine.</span><span class="sxs-lookup"><span data-stu-id="b2036-115">In short, it provides long-term agility.</span></span> <span data-ttu-id="b2036-116">I microservizi garantiscono una maggiore semplicità di gestione in sistemi grandi, complessi e altamente scalabili, poiché consentono di creare applicazioni basate su molti servizi distribuibili in modo indipendente, ciascuno con cicli di vita autonomi e specifici.</span><span class="sxs-lookup"><span data-stu-id="b2036-116">Microservices enable better maintainability in complex, large, and highly-scalable systems by letting you create applications based on many independently deployable services that each have granular and autonomous lifecycles.</span></span>

<span data-ttu-id="b2036-117">I microservizi, inoltre, possono essere ampliati in modo indipendente.</span><span class="sxs-lookup"><span data-stu-id="b2036-117">As an additional benefit, microservices can scale out independently.</span></span> <span data-ttu-id="b2036-118">In altre parole, non si ha più un'applicazione monolitica che deve essere ampliata come singola unità, ma è possibile ampliare solo determinati microservizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-118">Instead of having a single monolithic application that you must scale out as a unit, you can instead scale out specific microservices.</span></span> <span data-ttu-id="b2036-119">È possibile quindi ampliare solo l'area funzionale che richiede più potenza di elaborazione o larghezza di banda di rete per poter supportare la richiesta, senza dover ampliare inutilmente altre aree dell'applicazione.</span><span class="sxs-lookup"><span data-stu-id="b2036-119">That way, you can scale just the functional area that needs more processing power or network bandwidth to support demand, rather than scaling out other areas of the application that don't need to be scaled.</span></span> <span data-ttu-id="b2036-120">Ne consegue quindi una riduzione dei componenti hardware necessari e, pertanto, un considerevole risparmio economico.</span><span class="sxs-lookup"><span data-stu-id="b2036-120">That means cost savings because you need less hardware.</span></span>

<span data-ttu-id="b2036-126">La progettazione di applicazioni basate su microservizi con granularità fine consente processi di integrazione continua e recapito continuo.</span><span class="sxs-lookup"><span data-stu-id="b2036-126">Architecting fine-grained microservices-based applications enables continuous integration and continuous delivery practices.</span></span> <span data-ttu-id="b2036-127">Accelera inoltre la distribuzione di nuove funzioni nell'applicazione.</span><span class="sxs-lookup"><span data-stu-id="b2036-127">It also accelerates delivery of new functions into the application.</span></span> <span data-ttu-id="b2036-128">La composizione di applicazioni con granularità fine consente inoltre di eseguire e testare microservizi in isolamento ed evolverli in modo autonomo, pur mantenendo contratti precisi tra di essi.</span><span class="sxs-lookup"><span data-stu-id="b2036-128">Fine-grained composition of applications also allows you to run and test microservices in isolation, and to evolve them autonomously while maintaining clear contracts between them.</span></span> <span data-ttu-id="b2036-129">A condizione che non vengano apportate modifiche alle interfacce o ai contratti, è possibile modificare l'implementazione interna di qualsiasi microservizio o aggiungere nuove funzionalità senza interrompere gli altri microservizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-129">As long as you don't change the interfaces or contracts, you can change the internal implementation of any microservice or add new functionality without breaking other microservices.</span></span>

<span data-ttu-id="b2036-130">Di seguito sono elencati alcuni aspetti importanti per portare a termine correttamente la produzione di un sistema basato su microservizi:</span><span class="sxs-lookup"><span data-stu-id="b2036-130">The following are important aspects to enable success in going into production with a microservices-based system:</span></span>

- <span data-ttu-id="b2036-131">Monitoraggio e controlli di integrità dei servizi e dell'infrastruttura.</span><span class="sxs-lookup"><span data-stu-id="b2036-131">Monitoring and health checks of the services and infrastructure.</span></span>

- <span data-ttu-id="b2036-132">Infrastruttura scalabile per i servizi (ovvero, cloud e agenti di orchestrazione).</span><span class="sxs-lookup"><span data-stu-id="b2036-132">Scalable infrastructure for the services (that is, cloud and orchestrators).</span></span>

- <span data-ttu-id="b2036-133">Progettazione e implementazione della sicurezza a più livelli: autenticazione, autorizzazione, gestione dei segreti, comunicazioni protette e così via.</span><span class="sxs-lookup"><span data-stu-id="b2036-133">Security design and implementation at multiple levels: authentication, authorization, secrets management, secure communication, etc.</span></span>

- <span data-ttu-id="b2036-134">Distribuzione rapida delle applicazioni, in genere con più team di lavoro incentrati su diversi microservizi.</span><span class="sxs-lookup"><span data-stu-id="b2036-134">Rapid application delivery, usually with different teams focusing on different microservices.</span></span>

- <span data-ttu-id="b2036-135">Infrastruttura e processi DevOps e di integrazione continua/recapito continuo.</span><span class="sxs-lookup"><span data-stu-id="b2036-135">DevOps and CI/CD practices and infrastructure.</span></span>

<span data-ttu-id="b2036-136">Solo i prime tre aspetti sono descritti o introdotti nella presente guida.</span><span class="sxs-lookup"><span data-stu-id="b2036-136">Of these, only the first three are covered or introduced in this guide.</span></span> <span data-ttu-id="b2036-137">Gli ultimi due punti, correlati al ciclo di vita delle applicazioni, sono trattati nell'e-book [Ciclo di vita delle applicazioni Docker in contenitori con piattaforma e strumenti Microsoft](https://aka.ms/dockerlifecycleebook).</span><span class="sxs-lookup"><span data-stu-id="b2036-137">The last two points, which are related to application lifecycle, are covered in the additional [Containerized Docker Application Lifecycle with Microsoft Platform and Tools](https://aka.ms/dockerlifecycleebook) e-book.</span></span>
