## <a name="remarks"></a><span data-ttu-id="c1bec-115">Note</span><span class="sxs-lookup"><span data-stu-id="c1bec-115">Remarks</span></span>  
 <span data-ttu-id="c1bec-116">L'infrastruttura di affidabilità di common language runtime (CLR) distingue tra risorse e le interruzioni di errori di allocazione che si verificano nelle aree critiche di codice e quelli che si verificano in aree non critici del codice.</span><span class="sxs-lookup"><span data-stu-id="c1bec-116">The common language runtime (CLR) reliability infrastructure distinguishes between aborts and resource allocation failures that occur in critical regions of code and those that occur in non-critical regions of code.</span></span> <span data-ttu-id="c1bec-117">Questa distinzione è progettata per consentire agli host di impostare criteri diversi a seconda di dove si verifica un errore nel codice.</span><span class="sxs-lookup"><span data-stu-id="c1bec-117">This distinction is designed to allow hosts to set different policies depending on where a failure occurs in the code.</span></span>  
  
 <span data-ttu-id="c1bec-118">Oggetto *area critica del codice* qualsiasi spazio in cui CLR non può garantire che l'interruzione di un'attività o riesce a completare una richiesta di risorse influirà solo l'attività corrente.</span><span class="sxs-lookup"><span data-stu-id="c1bec-118">A *critical region of code* is any space where the CLR cannot guarantee that aborting a task or failing to complete a request for resources will affect only the current task.</span></span> <span data-ttu-id="c1bec-119">Ad esempio, se un'attività è bloccata da un blocco e riceve un valore HRESULT indicante un errore durante l'inoltro di una richiesta di allocazione della memoria, è sufficiente semplicemente interrompere l'attività per garantire la stabilità del <xref:System.AppDomain>, in quanto il <xref:System.AppDomain> potrebbe contenere altri attività in attesa per lo stesso blocco.</span><span class="sxs-lookup"><span data-stu-id="c1bec-119">For example, if a task is holding a lock and receives an HRESULT that indicates failure upon making a memory allocation request, it is insufficient simply to abort that task to ensure the stability of the <xref:System.AppDomain>, because the <xref:System.AppDomain> might contain other tasks waiting for the same lock.</span></span> <span data-ttu-id="c1bec-120">Per abbandonare corrente attività potrebbe causare altre attività smetterà di rispondere.</span><span class="sxs-lookup"><span data-stu-id="c1bec-120">To abandon the current task might cause those other tasks to stop responding.</span></span> <span data-ttu-id="c1bec-121">In tal caso, l'host deve essere in grado di scaricare l'intero <xref:System.AppDomain> invece di instabilità potenziali rischi.</span><span class="sxs-lookup"><span data-stu-id="c1bec-121">In such a case, the host needs the ability to unload the entire <xref:System.AppDomain> rather than risk potential instability.</span></span>  
  
 <span data-ttu-id="c1bec-122">Oggetto *area critica del codice*, d'altra parte, è un'area in cui CLR è in grado di garantire che un'interruzione o un errore interesserà solo l'attività su cui si verifica l'errore.</span><span class="sxs-lookup"><span data-stu-id="c1bec-122">A *non-critical region of code*, on the other hand, is a region where the CLR can guarantee that an abort or a failure will affect only the task upon which the error occurs.</span></span>  
  
 <span data-ttu-id="c1bec-123">CLR consente di distinguere anche tra interruzioni (non applicabili) normali e non normali.</span><span class="sxs-lookup"><span data-stu-id="c1bec-123">The CLR also distinguishes between graceful and non-graceful (rude) aborts.</span></span> <span data-ttu-id="c1bec-124">In generale, un'interruzione normali o irregolari si impegna per eseguire routine di gestione delle eccezioni e i finalizzatori prima dell'interruzione di un'attività, anche se questo tipo garantisce in alcun modo di un'interruzione non applicabili.</span><span class="sxs-lookup"><span data-stu-id="c1bec-124">In general, a normal or graceful abort makes every effort to run exception-handling routines and finalizers before aborting a task, while a rude abort makes no such guarantees.</span></span>
